# Context Engineering

Context engineering is the discipline of building dynamic systems that supply the right information and tools to a language model at the right time. Rather than crafting a single static prompt, you design pipelines that assemble context on the fly based on the current situation.

---

## Two Dimensions of Context

Context varies along two dimensions:

| Dimension   | Options                    | Description                                              |
|-------------|----------------------------|----------------------------------------------------------|
| **Mutability** | Static / Dynamic        | Does the context change during or between runs?          |
| **Lifetime**   | Runtime / Cross-conversation | Does the context exist only for one run or persist across many? |

---

## Three LangGraph Context Types

| Type                          | Mutability | Lifetime             | Mechanism                    | Example                             |
|-------------------------------|------------|----------------------|------------------------------|--------------------------------------|
| **Static runtime**            | Static     | Runtime              | `context` config parameter   | User ID, session metadata            |
| **Dynamic runtime (state)**   | Dynamic    | Runtime              | Graph state                  | Conversation messages, tool results  |
| **Dynamic cross-conversation (store)** | Dynamic | Cross-conversation | BaseStore (namespace/key)    | User preferences, learned facts      |

---

## Static Runtime Context

Static runtime context is set once at invocation time and remains constant throughout the run. Use it for user identity, feature flags, API keys, and other immutable metadata.

### Defining Context with a Dataclass

```python
from dataclasses import dataclass

@dataclass
class UserContext:
    user_id: str
    role: str
    permissions: list[str]
```

### Passing Context at Invocation

```python
config = {
    "configurable": {
        "context": UserContext(
            user_id="user-42",
            role="admin",
            permissions=["read", "write", "delete"]
        )
    }
}

result = graph.invoke(
    {"messages": [{"role": "user", "content": "Delete record 99"}]},
    config=config
)
```

### Accessing Context in a Prompt Middleware

A common pattern is using middleware to dynamically build the system prompt from context:

```python
from langgraph.prebuilt import create_react_agent

def dynamic_prompt(state, config):
    """Build a system prompt dynamically based on context."""
    context = config["configurable"]["context"]
    base_prompt = f"You are an assistant for user {context.user_id}."

    if context.role == "admin":
        base_prompt += " This user has admin privileges."

    base_prompt += f" Allowed actions: {', '.join(context.permissions)}."
    return [{"role": "system", "content": base_prompt}]

agent = create_react_agent(
    model=model,
    tools=tools,
    prompt=dynamic_prompt
)
```

### Accessing Context in Workflow Nodes

In a custom graph, nodes receive `config` as a parameter and can extract the context from it:

```python
def my_node(state, config):
    """A graph node that uses static context."""
    context = config["configurable"]["context"]
    user_id = context.user_id

    # Use context to customize behavior
    if context.role == "admin":
        return handle_admin_request(state)
    else:
        return handle_regular_request(state)
```

### Accessing Context in Tools via ToolRuntime

Tools access context through the `ToolRuntime` object. The generic type parameter provides type safety.

```python
from langchain.tools import tool, ToolRuntime

@tool
def delete_record(record_id: str, runtime: ToolRuntime[UserContext]) -> str:
    """Delete a record by ID."""
    context = runtime.context
    if "delete" not in context.permissions:
        return f"Error: User {context.user_id} lacks delete permission."
    # Perform deletion...
    return f"Record {record_id} deleted by {context.user_id}."
```

## Dynamic Runtime Context (State)

Dynamic runtime context changes during execution. The primary mechanism is the graph's **state**, which accumulates messages, tool results, and custom fields as the agent runs.

### Custom AgentState

Extend the default state to include domain-specific fields:

```python
from langgraph.graph import MessagesState
from typing import Optional

class AgentState(MessagesState):
    user_name: Optional[str] = None
    current_topic: Optional[str] = None
    retrieved_documents: list[str] = []
```

### Nodes Updating State

Any node can return updates to the state:

```python
def classify_topic(state):
    """Classify the conversation topic and update state."""
    messages = state["messages"]
    last_message = messages[-1].content

    topic = llm.invoke(
        f"Classify this message into a topic (billing, technical, general): {last_message}"
    ).content.strip()

    return {"current_topic": topic}
```

### Tools Updating State with Command

Tools can update dynamic state by returning `Command` objects:

```python
from langgraph.types import Command
from langchain.tools import tool

@tool
def set_topic(topic: str) -> Command:
    """Set the current conversation topic."""
    return Command(update={"current_topic": topic})

@tool
def store_documents(documents: list[str]) -> Command:
    """Store retrieved documents in state for later reference."""
    return Command(update={"retrieved_documents": documents})
```

## Dynamic Cross-Conversation Context (Store)

Cross-conversation context persists beyond a single thread. It is implemented through **stores** that use a namespace/key pattern. This is the mechanism for long-term memory.

### Setting Up a Store

```python
from langgraph.store.memory import InMemoryStore

store = InMemoryStore(
    index={
        "embed": embedding_function,
        "dims": 1536
    }
)

graph = builder.compile(store=store)
```

### Reading and Writing Store Data in Nodes

Nodes can access the store through the `store` parameter:

```python
def personalized_greeting(state, config, store):
    """Greet the user using information from long-term memory."""
    user_id = config["configurable"]["user_id"]

    # Read from store
    profile = store.get(("users",), user_id)
    if profile:
        name = profile.value.get("name", "there")
        greeting = f"Welcome back, {name}!"
    else:
        greeting = "Hello! I don't think we've met before."

    return {
        "messages": [{"role": "assistant", "content": greeting}]
    }
```

### Reading and Writing Store Data in Tools

Tools access the store through `ToolRuntime`:

```python
@tool
def save_preference(key: str, value: str, runtime: ToolRuntime) -> str:
    """Save a user preference for future sessions."""
    user_id = runtime.context.user_id
    store = runtime.store

    # Read existing preferences
    existing = store.get(("preferences", user_id), "prefs")
    prefs = existing.value if existing else {}

    # Update and save
    prefs[key] = value
    store.put(("preferences", user_id), "prefs", prefs)
    return f"Preference saved: {key} = {value}"
```

## Combining All Three Context Types

A well-designed agent typically uses all three context types together:

```python
from dataclasses import dataclass
from langgraph.graph import StateGraph, START, END
from langgraph.store.memory import InMemoryStore

@dataclass
class AppContext:
    user_id: str
    tenant_id: str

class AgentState(MessagesState):
    current_intent: str = ""

def router_node(state, config, store):
    """Route using static context, dynamic state, and long-term memory."""
    # Static runtime context
    ctx = config["configurable"]["context"]
    user_id = ctx.user_id

    # Dynamic cross-conversation context (store)
    history = store.search(("interactions", user_id), query=state["messages"][-1].content)
    past_intents = [h.value.get("intent") for h in history]

    # Dynamic runtime context (state)
    current_intent = state.get("current_intent", "")

    # Combine all context to make a routing decision
    if current_intent == "billing" or "billing" in past_intents:
        return {"current_intent": "billing"}
    return {"current_intent": "general"}

builder = StateGraph(AgentState)
builder.add_node("router", router_node)
builder.add_edge(START, "router")
builder.add_edge("router", END)

store = InMemoryStore()
graph = builder.compile(store=store)

result = graph.invoke(
    {"messages": [{"role": "user", "content": "Check my invoice"}]},
    config={
        "configurable": {
            "context": AppContext(user_id="u-1", tenant_id="t-1"),
            "thread_id": "thread-001"
        }
    }
)
```
