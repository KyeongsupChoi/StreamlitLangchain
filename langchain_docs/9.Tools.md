# Tools

Tools allow language models to interact with external systems, APIs, databases, and other resources. They extend agent capabilities beyond text generation by providing callable functions that the model can invoke when needed.

## Creating Tools

### The @tool Decorator

The simplest way to create a tool is with the `@tool` decorator. LangChain infers the tool's name from the function name, the description from the docstring, and the argument schema from the type hints.

```python
from langchain.tools import tool

@tool
def search_database(query: str, limit: int = 10) -> str:
    """Search the customer database for records matching the query.
    Args:
        query: Search terms to look for
        limit: Maximum number of results to return
    """
    return f"Found {limit} results for '{query}'"
```

### Custom Name and Description

Override the default name and description by passing arguments to the decorator:

```python
@tool("web_search")
def search(query: str) -> str:
    """Search the web for information."""
    return f"Results for: {query}"
```

```python
@tool("calculator", description="Performs arithmetic calculations.")
def calc(expression: str) -> str:
    """Evaluate a math expression."""
    return str(eval(expression))
```

### Pydantic and JSON Schema for Arguments

For more control over argument validation and descriptions, use a Pydantic model as the `args_schema`:

```python
from pydantic import BaseModel, Field
from typing import Literal

class WeatherInput(BaseModel):
    location: str = Field(description="City name or coordinates")
    units: Literal["celsius", "fahrenheit"] = Field(default="celsius")
    include_forecast: bool = Field(default=False)

@tool(args_schema=WeatherInput)
def get_weather(location: str, units: str = "celsius", include_forecast: bool = False) -> str:
    """Get current weather and optional forecast."""
    base = f"Weather for {location} in {units}: 22 degrees"
    if include_forecast:
        base += " | Forecast: Sunny for the next 3 days"
    return base
```

### Reserved Arguments

Certain parameter names are reserved and automatically injected by the runtime rather than being passed by the model:

| Parameter Name | Type             | Description                                                  |
|----------------|------------------|--------------------------------------------------------------|
| `config`       | `RunnableConfig` | Runtime configuration (callbacks, tags, metadata, etc.)      |
| `runtime`      | `ToolRuntime`    | Access to state, context, store, and stream writer           |

These parameters are hidden from the model's schema. The model never sees or fills them.

```python
from langchain_core.runnables import RunnableConfig
from langchain.tools import tool

@tool
def my_tool(query: str, config: RunnableConfig) -> str:
    """A tool that uses runtime config."""
    user_id = config.get("configurable", {}).get("user_id", "unknown")
    return f"Running query '{query}' for user {user_id}"
```

## Accessing Context via ToolRuntime

`ToolRuntime` provides tools with access to agent state, immutable context, persistent storage, and a stream writer. Declare `runtime: ToolRuntime` as a parameter and it is automatically injected.

### State (Short-Term Memory)

State represents the current conversation and any custom fields in your graph's state. Tools can read state and update it using `Command`.

**Reading state:**

```python
from langchain.tools import tool, ToolRuntime
from langchain_core.messages import HumanMessage

@tool
def get_last_user_message(runtime: ToolRuntime) -> str:
    """Get the most recent message from the user."""
    messages = runtime.state["messages"]
    for message in reversed(messages):
        if isinstance(message, HumanMessage):
            return message.content
    return "No user messages found"
```

**Updating state with Command:**

Tools can update state by returning a `Command` object instead of a plain string. This lets tools write back to the graph state.

```python
from langgraph.types import Command

@tool
def set_user_name(new_name: str) -> Command:
    """Set the user's name in the conversation state."""
    return Command(update={"user_name": new_name})
```

### Context (Immutable Configuration)

Context provides read-only information that is set at invocation time and does not change during the run. Typical uses include user IDs, session metadata, and environment flags. Use a dataclass for type safety.

```python
from dataclasses import dataclass

@dataclass
class UserContext:
    user_id: str

@tool
def get_account_info(runtime: ToolRuntime[UserContext]) -> str:
    """Get the current user's account information."""
    user_id = runtime.context.user_id
    return f"Account info for user {user_id}: Premium tier, joined 2023-01-15"
```

Pass the context when invoking the graph:

```python
result = graph.invoke(
    {"messages": [{"role": "user", "content": "Show my account"}]},
    config={"configurable": {"context": UserContext(user_id="user-123")}}
)
```

### Store (Long-Term Persistent Memory)

The store provides cross-session persistence using a namespace/key pattern. It survives beyond a single conversation thread.

```python
@tool
def save_user_info(user_id: str, user_info: dict, runtime: ToolRuntime) -> str:
    """Save user info."""
    store = runtime.store
    store.put(("users",), user_id, user_info)
    return "Successfully saved user info."
```

```python
@tool
def get_user_info(user_id: str, runtime: ToolRuntime) -> str:
    """Look up user info."""
    store = runtime.store
    item = store.get(("users",), user_id)
    if item:
        return f"User info: {item.value}"
    return "No user info found."
```

### Stream Writer for Real-Time Updates

The stream writer lets tools send intermediate updates to the client before the final result is ready. This is useful for long-running tools where you want to show progress.

```python
@tool
def get_weather(city: str, runtime: ToolRuntime) -> str:
    """Get weather for a given city."""
    writer = runtime.stream_writer
    writer(f"Looking up data for city: {city}")
    # ... perform actual lookup ...
    writer(f"Processing weather data...")
    return f"It's always sunny in {city}!"
```

## ToolNode

`ToolNode` is a prebuilt LangGraph node that executes tools requested by the model. It reads `AIMessage` tool calls from the state and runs the corresponding tools, returning `ToolMessage` results.

### Basic Setup

```python
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.graph import StateGraph, MessagesState, START, END

tools = [search_database, get_weather]
tool_node = ToolNode(tools)

builder = StateGraph(MessagesState)
builder.add_node("agent", call_model)
builder.add_node("tools", tool_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", tools_condition)
builder.add_edge("tools", "agent")
graph = builder.compile()
```

### Routing with tools_condition

`tools_condition` is a helper that checks whether the last message contains tool calls. If it does, it routes to the `"tools"` node; otherwise it routes to `END`.

```python
builder.add_conditional_edges(
    "agent",
    tools_condition,
    # Implicit mapping: returns "tools" if tool calls present, else END
)
```

### Error Handling

By default, tool errors propagate. You can enable automatic error handling so the model receives the error as a `ToolMessage` and can retry.

**Boolean flag (returns error text to model):**

```python
tool_node = ToolNode(tools, handle_tool_errors=True)
```

**Custom error handler:**

```python
def my_error_handler(error: Exception, tool_call: dict) -> str:
    return f"Tool '{tool_call['name']}' failed: {str(error)}. Please try a different approach."

tool_node = ToolNode(tools, handle_tool_errors=my_error_handler)
```

**Handling specific exceptions:**

```python
tool_node = ToolNode(
    tools,
    handle_tool_errors={
        ValueError: "Invalid input provided. Please check your arguments.",
        TimeoutError: "The operation timed out. Try again with simpler parameters.",
    }
)
```

### State Injection

ToolNode automatically injects `runtime` and `config` into tools that declare those parameters. No extra configuration is needed. The tool simply declares the parameter and ToolNode fills it.

## Server-Side Tool Use

When deploying agents as services, tools run on the server. Key considerations:

- **Authentication**: Validate user credentials before executing tools that access protected resources. Use the `config` or `context` to pass authenticated user information.
- **Rate limiting**: Protect external API calls from excessive invocation.
- **Sandboxing**: Isolate code execution tools to prevent security issues.
- **Timeouts**: Set appropriate timeouts for external calls to prevent hanging.

```python
@tool
def secure_database_query(query: str, config: RunnableConfig) -> str:
    """Query the database with the authenticated user's permissions."""
    user_id = config["configurable"]["user_id"]
    permissions = get_user_permissions(user_id)
    if not permissions.can_query:
        return "Error: You do not have permission to query the database."
    return execute_query(query, permissions)
```
