## LangChain project checklist

### Models
- [x] Choose a **chat model provider + model identifier** (static), or implement **dynamic model selection** at runtime.
- [x] Use a consistent model interface (`init_chat_model` or a provider class) and support:
  - [x] `invoke()` for normal calls
  - [x] `stream()` for progressive UX
  - [x] `batch()` / `batch_as_completed()` for throughput
- [x] Define and document **standard model parameters** you rely on:
  - [x] `model`, `api_key`, `temperature`, `max_tokens`, `timeout`, `max_retries`
- [x] Handle **token usage** tracking with context manager.
- [x] Decide whether your app needs **multimodal** support (content blocks).
- [x] Decide whether you need **reasoning output** surfaced (and at what level).
- [x] Add **rate limiting** if you expect bursts or background jobs.
- [x] Document any **base_url / proxy** requirements (AWS Bedrock endpoints):
  - [ ] **AWS Bedrock configuration**:
    - [ ] Set `region_name` (e.g., `us-east-1`, `us-west-2`)
    - [ ] Configure AWS credentials (one of):
      - [ ] Environment variables: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`
      - [ ] AWS credentials file (`~/.aws/credentials`)
      - [ ] IAM role (for EC2/ECS/Lambda execution)
      - [ ] AWS SSO profile
    - [ ] Specify `model_id` for Bedrock models (e.g., `anthropic.claude-3-sonnet-20240229-v1:0`)
    - [ ] Handle cross-region inference if needed (requires specific model ARNs)
  - [ ] **Corporate proxy** configuration (if behind firewall):
    - [ ] Set boto3 proxy via `botocore.config.Config(proxies={'https': 'proxy-url'})`
    - [ ] Or use environment variables: `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY`
    - [ ] Handle proxy authentication if required
  - [ ] **Connection parameters** for AWS endpoints:
    - [ ] `connect_timeout`, `read_timeout`: configure via botocore Config
    - [ ] `max_attempts`: retry logic for API throttling (default 3)
    - [ ] `retry_mode`: standard/adaptive for handling rate limits
  - [ ] Document **environment-specific configurations**:
    - [ ] Different AWS regions per environment (dev/staging/prod)
    - [ ] Different IAM roles/credentials per environment
    - [ ] Model availability by region

### Messages
- [ ] Understand **message fundamentals** (role, content, metadata):
  - [ ] Know the three core components of messages (role, content, metadata)
  - [ ] Understand that messages work consistently across all model providers
- [ ] Choose appropriate **message format** for your use case:
  - [ ] **Text prompts** for:
    - [ ] Single, standalone requests
    - [ ] No conversation history needed
    - [ ] Minimal code complexity
  - [ ] **Message prompts** for:
    - [ ] Multi-turn conversations
    - [ ] Multimodal content (images, audio, files)
    - [ ] System instructions and context
  - [ ] **Dictionary format** for OpenAI-compatible APIs
- [ ] Implement **message types** correctly:
  - [ ] **SystemMessage**: Define model behavior and role
    - [ ] Set tone and personality
    - [ ] Establish response guidelines
    - [ ] Provide context and constraints
  - [ ] **HumanMessage**: Represent user input
    - [ ] Support text content
    - [ ] Add optional metadata (name, id) for tracking
    - [ ] Handle multimodal content if needed
  - [ ] **AIMessage**: Handle model responses
    - [ ] Access `.text` for simple content
    - [ ] Access `.content` for raw content
    - [ ] Process `.tool_calls` when using tools
    - [ ] Check `.usage_metadata` for token counts
    - [ ] Review `.response_metadata` for provider-specific data
  - [ ] **ToolMessage**: Return tool execution results
    - [ ] Match `tool_call_id` from AIMessage
    - [ ] Provide stringified tool output in `content`
    - [ ] Use `artifact` field for supplementary data not sent to model
- [ ] Handle **message metadata** appropriately:
  - [ ] Use `name` field to identify different users/speakers
  - [ ] Use `id` field for tracing and debugging
  - [ ] Track `usage_metadata` for token counting and cost management
  - [ ] Access `response_metadata` for provider-specific information
- [ ] Work with **message content** correctly:
  - [ ] Understand `content` attribute (loosely-typed: string or list)
  - [ ] Use `content_blocks` property for type-safe standard representation
  - [ ] Choose between provider-native format vs standard content blocks
- [ ] Implement **multimodal content** if needed:
  - [ ] **Image content**: support URL, base64, or file_id formats
  - [ ] **Audio content**: support base64 or file_id formats
  - [ ] **Video content**: support base64 or file_id formats
  - [ ] **File content**: support PDF and documents (URL, base64, file_id)
  - [ ] Specify correct MIME types for all media
  - [ ] Check model provider support for each media type
- [ ] Understand **content block types**:
  - [ ] **Core blocks**:
    - [ ] TextContentBlock for standard text output
    - [ ] ReasoningContentBlock for model reasoning steps
  - [ ] **Multimodal blocks**:
    - [ ] ImageContentBlock, AudioContentBlock, VideoContentBlock
    - [ ] FileContentBlock for PDFs and documents
    - [ ] PlainTextContentBlock for .txt and .md files
  - [ ] **Tool calling blocks**:
    - [ ] ToolCall for function calls
    - [ ] ToolCallChunk for streaming tool calls
    - [ ] InvalidToolCall for error handling
  - [ ] **Server-side tool blocks** (if using server-side execution):
    - [ ] ServerToolCall, ServerToolCallChunk, ServerToolResult
- [ ] Handle **streaming messages** correctly:
  - [ ] Process `AIMessageChunk` objects during streaming
  - [ ] Combine chunks into full messages (using `+` operator)
  - [ ] Handle tool call chunks that accumulate into full calls
- [ ] Manage **conversation state** with messages:
  - [ ] Build message lists that alternate HumanMessage and AIMessage
  - [ ] Include SystemMessage at the beginning when needed
  - [ ] Insert ToolMessage after AIMessage with tool_calls
  - [ ] Ensure message history remains valid for provider requirements

### Tool calling
- [x] Define tools with **clear names, descriptions, and schemas** (they become part of the model's prompt):
  - [x] **Tool naming**:
    - [x] Use verb_noun format (e.g., `get_weather`, `search_database`, `send_email`)
    - [x] Be specific and descriptive (avoid generic names like `tool1`, `helper`)
    - [x] Use snake_case for consistency with LangChain conventions
    - [x] Keep names concise but unambiguous (model sees these in prompt)
  - [x] **Tool descriptions**:
    - [x] Write clear, imperative descriptions explaining what the tool does
    - [x] Include when/why to use the tool (helps model decision-making)
    - [x] Specify any constraints or limitations (API rate limits, data freshness, etc.)
    - [x] Keep descriptions concise but complete (they consume prompt tokens)
  - [ ] **Schema design**:
    - [ ] Use Pydantic models or `@tool` decorator for automatic schema generation
    - [ ] Define all required vs optional parameters explicitly
    - [ ] Add field descriptions for each parameter (shown to model)
    - [ ] Use appropriate types (str, int, float, bool, enums, lists, nested objects)
    - [ ] Set sensible defaults for optional parameters
    - [ ] Add validation constraints (min/max, regex patterns, allowed values)
  - [ ] **Parameter documentation**:
    - [ ] Describe format expectations (e.g., "ISO 8601 date", "valid email address")
    - [ ] Provide examples in descriptions when format is complex
    - [ ] Clarify units for numeric parameters (seconds, meters, USD, etc.)
    - [ ] Document relationships between parameters if applicable
  - [ ] **Return value specification**:
    - [ ] Document what the tool returns (type and structure)
    - [ ] Explain possible error messages or failure modes
    - [ ] Indicate if tool has side effects (writes data, sends notifications)
- [ ] Bind tools with `bind_tools(...)` (and optionally):
  - [ ] force tool use via `tool_choice`
  - [ ] enable/disable `parallel_tool_calls`
- [ ] Implement the **tool execution loop** correctly:
  - [ ] model produces tool calls
  - [ ] execute tool(s)
  - [ ] return results as `ToolMessage` with matching `tool_call_id`
  - [ ] re-invoke model with updated message history
- [ ] Support **streaming tool calls** if you stream responses (tool call chunks accumulate into full calls).

### Tool runtime & context (see 9.Tools.md)
- [ ] Decide if tools need **runtime context** via `ToolRuntime`:
  - [ ] **State access**: read conversation history and custom state fields
  - [ ] **State updates**: return `Command` objects to mutate agent state
  - [ ] **Context** (immutable): user IDs, session details via typed `@dataclass` schema
  - [ ] **Store** (long-term memory): persistent data across conversations via namespace/key
  - [ ] **Stream writer**: emit real-time progress updates from long-running tools
- [ ] Decide if you need **ToolNode** for LangGraph workflows:
  - [ ] Configure error handling strategy (boolean, custom handler, exception types)
  - [ ] Use `tools_condition` for conditional routing based on tool calls
  - [ ] Enable state injection for tools that need graph state
- [ ] Evaluate **prebuilt tools** (search, code execution, databases) before writing custom ones.
- [ ] Decide if you need **server-side tool use** (provider-hosted web search, code interpreters).

  - [ ] TypedDict (lightweight, no runtime validation)
  - [ ] JSON Schema (interop/control)
- [ ] Choose/confirm the **structured output method** strategy:
  - [ ] provider-native (`json_schema`-style)
  - [ ] tool/function-calling derived structure
  - [ ] json-mode style (prompt-based constraints)
- [ ] Decide whether you need `include_raw=True` (store raw message + parsed + error).

### Agents
- [ ] Decide whether you need an **agent loop** (vs direct model calls).
- [ ] Define the **stop condition** (final answer vs iteration limit).

- [ ] Define tools as either:

  - [ ] via middleware state schema

### Memory & conversation state (see 10.Memory_concepts.md)
- [x] Decide on **short-term memory** approach (message state).
- [ ] Decide on **persistence** (in-memory vs durable store) if conversations must survive restarts.
- [ ] Standardize identifiers like **thread/conversation IDs**.
- [ ] Choose **long-term memory** type if needed:
  - [ ] **Semantic memory** (facts about users/entities):
    - [ ] Profile approach (single updated JSON document) vs collection approach (discrete items)
  - [ ] **Episodic memory** (past experiences as few-shot examples)
  - [ ] **Procedural memory** (self-modifying system prompts via reflection)
- [ ] Choose **memory writing strategy**:
  - [ ] Hot path (real-time during conversation, immediate availability, adds latency)
  - [ ] Background (post-conversation, no latency impact, delayed availability)
- [ ] Choose **memory storage backend**:
  - [ ] `InMemoryStore` (development/testing)
  - [ ] `PostgresStore` or database-backed (production)
  - [ ] Configure semantic search with embeddings if using collection approach

### Context engineering (see 11.Context_engineering.md)
- [ ] Identify which **context types** your app needs:
  - [ ] **Static runtime context**: immutable metadata passed at invocation (user ID, session, config)
    - [ ] Define `@dataclass` context schema
    - [ ] Pass via `context=` argument to `invoke`/`stream`
    - [ ] Access in tools via `ToolRuntime`, in nodes via `Runtime`, in prompts via middleware
  - [ ] **Dynamic runtime context** (state): mutable data evolving during a single run
    - [ ] Extend `AgentState` with custom fields
    - [ ] Use reducers for fields updated by parallel tool calls
  - [ ] **Dynamic cross-conversation context** (store): persistent data across sessions
    - [ ] Configure store with namespace/key hierarchy
    - [ ] Read/write from tools and nodes

### RAG - Retrieval-Augmented Generation (see 12.RAG_agent.md)
- [ ] Decide if your app needs **RAG** (answering questions over custom data).
- [ ] **Indexing pipeline**:
  - [ ] Choose **document loaders** (PDF, web, CSV, databases, etc.)
  - [ ] Choose **text splitter** and configure chunk size / overlap (`RecursiveCharacterTextSplitter`)
  - [ ] Choose **embedding model** provider (OpenAI, Cohere, HuggingFace, etc.)
  - [ ] Choose **vector store** (InMemory, Chroma, FAISS, Pinecone, PGVector, Qdrant, etc.)

  - [ ] **RAG agent** (tool-based): model decides when and how to search

    - [ ] Support iterative retrieval (multiple searches per query)
  - [ ] **RAG chain** (two-step): always retrieve, single inference call, lower latency
    - [ ] Inject context via `dynamic_prompt` middleware
- [ ] Decide whether to **return source documents** alongside answers.

### Multi-agent: supervisor / subagents (see 16.Multi_agent_subagents.md)

  - [ ] Create **specialized sub-agents** with domain-specific tools and prompts
  - [ ] Wrap sub-agents as **tools** for the supervisor (`@tool` returning sub-agent output)
  - [ ] Write clear tool descriptions so supervisor knows when to delegate
  - [ ] Create **supervisor agent** with only high-level tools
- [ ] Handle **multi-domain requests** (supervisor calls multiple sub-agents in sequence).
- [ ] Add **human-in-the-loop** on sensitive sub-agent actions (email sends, calendar changes).
- [ ] Control **information flow**:
  - [ ] Pass conversation context to sub-agents via `ToolRuntime.state`

### Multi-agent: handoffs / state machine (see 17.Multi_agent_handoffs.md)
- [ ] Decide if your workflow has **sequential stages** (customer support, onboarding, etc.).
- [ ] **State machine pattern** (single agent, dynamic behavior):
  - [ ] Define custom state with `current_step` field
  - [ ] Create **step configuration** mapping steps to prompts, tools, and required fields
  - [ ] Implement **state-transition tools** returning `Command` with step updates
  - [ ] Use `@wrap_model_call` middleware to apply step-specific config dynamically
- [ ] Support **backward transitions** (user corrections, escalation back to earlier steps).
- [ ] Add **message summarization** middleware if conversations grow long.

### Multi-agent: router (see 18.Multi_agent_router.md)
- [ ] Decide if your app queries **multiple independent knowledge sources** in parallel.
- [ ] **Router pattern** (classify -> parallel dispatch -> synthesize):
  - [ ] Define state schemas: `AgentInput`, `AgentOutput`, `Classification`, `RouterState`
  - [ ] Use **reducers** (`operator.add`) to collect parallel agent outputs
  - [ ] Implement **classification** with structured output (Pydantic model)
  - [ ] Create **specialized agents** per knowledge source (GitHub, Notion, Slack, etc.)
  - [ ] Use **Send API** for parallel fan-out to selected agents
  - [ ] Implement **synthesis node** to combine results into coherent answer
- [ ] Build with `StateGraph` and `add_conditional_edges`.
- [ ] For multi-turn conversations, wrap stateless router as a **tool** in a conversational agent.

### Multi-agent: progressive disclosure / skills (see 19.Multi_agent_skills.md)
- [ ] Decide if your agent has **too many skills to fit in context** upfront.
- [ ] **Skills pattern** (on-demand context loading):
  - [ ] Define skills as structured objects (name, short description, full content)
  - [ ] Implement `load_skill` tool for on-demand retrieval
  - [ ] Use **middleware** to inject skill descriptions into system prompt
  - [ ] Track loaded skills in agent state
- [ ] Choose **storage backend** (in-memory dict, file system, database, vector store).
- [ ] Choose **discovery method** (system prompt listing, semantic search, LLM classification).
- [ ] Choose **disclosure strategy** (lazy on-demand, eager pre-classified, hybrid).
- [ ] Keep skills **under 2K tokens**; split large ones into sub-skills.

### LangGraph: Graph API (see 20.Graph_API.md)
- [ ] Decide whether to use **Graph API** (explicit graph structure) vs **Functional API** (Python control flow).
- [ ] **State definition**:
  - [ ] Choose schema type: `TypedDict`, `dataclass`, or Pydantic `BaseModel`
  - [ ] Define **reducers** for fields updated by parallel nodes (`Annotated[list, operator.add]`)
  - [ ] Use `MessagesState` for standard chat patterns
  - [ ] Consider **multiple schemas** (InputState, OverallState, OutputState) for clean interfaces
- [ ] **Nodes**:
  - [ ] Define node functions accepting (state, config, runtime)
  - [ ] Use `START` and `END` special nodes
  - [ ] Enable **node caching** for expensive operations (`CachePolicy(ttl=...)`)
- [ ] **Edges**:
  - [ ] Normal edges for unconditional transitions
  - [ ] Conditional edges with routing functions for branching
  - [ ] Parallel edges for fan-out execution
  - [ ] **Send API** for dynamic routing with unknown edge counts
- [ ] Use **Command** object when combining state updates with routing.
- [ ] **Compile** graph before use and configure:
  - [ ] Checkpointer for persistence
  - [ ] Store for long-term memory
  - [ ] Context schema for runtime config
  - [ ] Recursion limit (default 1000, adjust as needed)
  - [ ] `RemainingSteps` for graceful degradation near limit

### LangGraph: Functional API (see 21.Functional_API.md)
- [ ] Decide if **Functional API** fits better (standard Python control flow, less boilerplate).
- [ ] Use `@entrypoint` for **workflow entry points**:
  - [ ] Single positional argument (use dict for multiple inputs)
  - [ ] Injectable params: `previous` (short-term memory), `store`, `writer`, `config`
  - [ ] Inputs and outputs must be **JSON-serializable**
- [ ] Use `@task` for **discrete units of work**:
  - [ ] Call `.result()` (sync) or `await` (async) for task output
  - [ ] Tasks are checkpointed automatically (resume without recomputation)
  - [ ] Run multiple tasks in parallel for I/O-bound operations
- [ ] Use `interrupt()` for **human-in-the-loop** pauses; resume with `Command(resume=...)`.
- [ ] Use `entrypoint.final` to **decouple return value from checkpoint save**.
- [ ] **Determinism**: encapsulate all randomness (API calls, timestamps) inside `@task` for correct HITL resume.
- [ ] **Idempotency**: design tasks to be safe for re-execution (use upserts, check existing state).
- [ ] Avoid **common pitfalls**:
  - [ ] Side effects outside tasks (will replay on resume)
  - [ ] Non-deterministic control flow (time-dependent branching)
  - [ ] Non-serializable inputs/outputs
