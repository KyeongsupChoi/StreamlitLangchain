# LangGraph Functional API

The Functional API provides an alternative to the Graph API for building stateful workflows. Instead of defining nodes and edges, you write regular Python functions decorated with `@entrypoint` and `@task`. The runtime handles checkpointing, resumption, and parallelism.

---

## Core Decorators

| Decorator       | Purpose                                                       |
|-----------------|---------------------------------------------------------------|
| `@entrypoint`   | Defines the top-level workflow function (the graph equivalent) |
| `@task`          | Defines a checkpointable unit of work within an entrypoint    |

---

## Comparison: Functional vs Graph API

| Feature                  | Functional API                        | Graph API                              |
|--------------------------|---------------------------------------|----------------------------------------|
| **Definition style**     | Decorated Python functions            | StateGraph with nodes and edges        |
| **State management**     | Implicit (via `previous` parameter)   | Explicit (TypedDict/Pydantic state)    |
| **Control flow**         | Standard Python (if/for/while)        | Conditional edges and routing functions |
| **Parallelism**          | Multiple `task()` calls + `.result()` | `Send` API or parallel edges           |
| **Checkpointing**        | Automatic at task boundaries          | Automatic at node boundaries           |
| **Human-in-the-loop**    | `interrupt()` inside entrypoint       | `interrupt()` inside nodes             |
| **Best for**             | Linear/branching workflows            | Complex cyclic graphs, multi-agent     |

---

## Example: Workflow with Human Review

```python
from langgraph.func import entrypoint, task
from langgraph.types import interrupt, Command
from langgraph.checkpoint.memory import MemorySaver

@task
def draft_essay(topic: str) -> str:
    """Generate an essay draft."""
    response = llm.invoke([
        {"role": "system", "content": "Write a short essay on the given topic."},
        {"role": "user", "content": topic}
    ])
    return response.content

@task
def revise_essay(draft: str, feedback: str) -> str:
    """Revise an essay based on feedback."""
    response = llm.invoke([
        {"role": "system", "content": "Revise this essay based on the feedback provided."},
        {"role": "user", "content": f"Essay:\n{draft}\n\nFeedback:\n{feedback}"}
    ])
    return response.content

@entrypoint(checkpointer=MemorySaver())
def essay_workflow(topic: str) -> str:
    """Write an essay with human review."""
    # Step 1: Draft
    draft = draft_essay(topic).result()

    # Step 2: Human review (pauses execution)
    feedback = interrupt({
        "draft": draft,
        "prompt": "Please review this draft and provide feedback, or type 'approve' to accept."
    })

    # Step 3: Revise or return
    if feedback.lower() == "approve":
        return draft
    else:
        revised = revise_essay(draft, feedback).result()
        return revised
```

### Running the Workflow

```python
config = {"configurable": {"thread_id": "essay-1"}}

# Start the workflow -- will pause at interrupt()
result = essay_workflow.invoke("The future of AI", config=config)
# result contains the interrupt payload with the draft

# Resume with feedback
result = essay_workflow.invoke(
    Command(resume="Make it more concise and add a conclusion"),
    config=config
)
# result contains the revised essay
```

## Entrypoint

### Definition

`@entrypoint` marks a function as a workflow entry point. It is the Functional API equivalent of a compiled graph.

```python
from langgraph.func import entrypoint
from langgraph.checkpoint.memory import MemorySaver

@entrypoint(checkpointer=MemorySaver())
def my_workflow(input_data: str) -> str:
    """A simple workflow."""
    result = some_task(input_data).result()
    return result
```

### Injectable Parameters

Entrypoint functions can declare special parameters that are automatically injected:

| Parameter   | Type              | Description                                    |
|-------------|-------------------|------------------------------------------------|
| `previous`  | Any               | The return value from the previous invocation (short-term memory) |
| `store`     | `BaseStore`       | Long-term persistent memory store              |
| `writer`    | `StreamWriter`    | For streaming intermediate results             |
| `config`    | `RunnableConfig`  | Runtime configuration                          |

```python
@entrypoint(checkpointer=MemorySaver())
def my_workflow(
    user_message: str,
    previous: list = [],
    store: BaseStore = None,
    config: RunnableConfig = None,
    writer: StreamWriter = None,
) -> list:
    """Workflow with all injectable parameters."""
    # previous contains the return value from the last call with this thread_id
    history = previous or []
    history.append({"role": "user", "content": user_message})

    # Use writer for streaming
    writer("Processing your request...")

    # Use store for long-term memory
    user_id = config["configurable"]["user_id"]
    memories = store.search(("memories", user_id), query=user_message)

    response = process_with_context(user_message, history, memories)
    history.append({"role": "assistant", "content": response})

    return history
```

### Execution

```python
# Invoke (synchronous)
result = my_workflow.invoke("Hello", config={"configurable": {"thread_id": "t1"}})

# Stream
for event in my_workflow.stream("Hello", config={"configurable": {"thread_id": "t1"}}):
    print(event)
```

### Resuming After Interrupt

```python
# Resume with a value
result = my_workflow.invoke(
    Command(resume="user's response"),
    config={"configurable": {"thread_id": "t1"}}
)
```

### Error Recovery

If a workflow fails mid-execution, it can be retried. Tasks that already completed will return their cached results (from checkpoints) rather than re-executing.

```python
try:
    result = my_workflow.invoke("input", config=config)
except Exception:
    # Fix the issue, then retry -- completed tasks are not re-run
    result = my_workflow.invoke("input", config=config)
```

## Short-Term Memory with `previous`

The `previous` parameter provides the return value from the last successful invocation with the same `thread_id`. This is the Functional API's mechanism for short-term (conversation) memory.

```python
@entrypoint(checkpointer=MemorySaver())
def chatbot(user_message: str, previous: list = []) -> list:
    """A chatbot with conversation memory."""
    messages = previous or []
    messages.append({"role": "user", "content": user_message})

    response = llm.invoke(messages)
    messages.append({"role": "assistant", "content": response.content})

    return messages

config = {"configurable": {"thread_id": "chat-1"}}

# Turn 1
result = chatbot.invoke("My name is Alice", config=config)
# result = [{"role": "user", ...}, {"role": "assistant", ...}]

# Turn 2 -- previous contains the result from Turn 1
result = chatbot.invoke("What's my name?", config=config)
# The agent knows the name is Alice from previous
```

## entrypoint.final

`entrypoint.final` decouples the return value from the value saved to the checkpoint. This is useful when you want to return a response to the user but save different data for the `previous` parameter.

```python
from langgraph.func import entrypoint

@entrypoint(checkpointer=MemorySaver())
def my_workflow(user_message: str, previous: dict = {}) -> entrypoint.final[str, dict]:
    """Return a string to the user but save a dict to the checkpoint."""
    history = previous.get("history", [])
    history.append(user_message)

    response = llm.invoke(history).content

    # Return type: entrypoint.final[return_type, save_type]
    # First value is returned to caller, second is saved as 'previous'
    return entrypoint.final(
        value=response,                          # Returned to the caller
        save={"history": history, "turns": len(history)}  # Saved as 'previous'
    )
```

## Task

### Definition

`@task` marks a function as a checkpointable unit of work. When called inside an entrypoint, it returns a **future** object.

```python
from langgraph.func import task

@task
def analyze_sentiment(text: str) -> str:
    """Analyze the sentiment of text."""
    response = llm.invoke([
        {"role": "system", "content": "Classify sentiment as positive, negative, or neutral."},
        {"role": "user", "content": text}
    ])
    return response.content
```

### Execution

**Synchronous** -- call `.result()` to block and get the value:

```python
@entrypoint(checkpointer=MemorySaver())
def my_workflow(text: str) -> str:
    future = analyze_sentiment(text)
    result = future.result()  # Blocks until complete
    return result
```

**Async** -- use `await` directly:

```python
@entrypoint(checkpointer=MemorySaver())
async def my_workflow(text: str) -> str:
    result = await analyze_sentiment(text)
    return result
```

### Future Objects

Calling a task returns a future. Multiple futures can execute in parallel:

```python
@entrypoint(checkpointer=MemorySaver())
def parallel_analysis(texts: list[str]) -> list[str]:
    """Analyze multiple texts in parallel."""
    # Launch all tasks (they run in parallel)
    futures = [analyze_sentiment(text) for text in texts]

    # Collect results (blocks until all complete)
    results = [f.result() for f in futures]

    return results
```

## When to Use Tasks

Use `@task` when you need:

1. **Checkpointing**: The result is saved, so if the workflow is retried, completed tasks are not re-run
2. **Human-in-the-loop**: Tasks before an `interrupt()` are not re-executed on resume
3. **Parallel execution**: Multiple task calls run concurrently
4. **Observability**: Each task appears as a distinct step in LangSmith traces
5. **Retries**: Failed tasks can be retried independently

```python
@task(retry_policy={"max_attempts": 3, "backoff_factor": 2})
def unreliable_api_call(query: str) -> str:
    """Call an API that sometimes fails."""
    return external_api.search(query)
```

---

## Serialization Requirements

Task inputs and outputs must be serializable (JSON-compatible) because they are saved to the checkpoint. This means:

- Use primitive types (str, int, float, bool, None)
- Use lists and dicts (not sets or custom objects)
- Pydantic models are supported
- Functions, generators, and file handles are not serializable

```python
# Good: serializable
@task
def good_task(query: str) -> dict:
    return {"answer": "...", "confidence": 0.95}

# Bad: not serializable
@task
def bad_task(query: str) -> object:
    return SomeCustomClass()  # Will fail at checkpoint
```

---

## Determinism Requirements for HITL

When using `interrupt()` for human-in-the-loop, the code path before the interrupt must be **deterministic**. On resume, the entrypoint re-executes from the beginning, replaying cached task results. If the code path differs, the replay will not match.

```python
# GOOD: Deterministic
@entrypoint(checkpointer=MemorySaver())
def good_workflow(input: str) -> str:
    step1 = task_a(input).result()        # Replayed from cache on resume
    feedback = interrupt({"draft": step1}) # Pauses here
    step2 = task_b(step1, feedback).result()
    return step2

# BAD: Non-deterministic (random choice changes on replay)
@entrypoint(checkpointer=MemorySaver())
def bad_workflow(input: str) -> str:
    import random
    if random.random() > 0.5:             # Different result on replay!
        step1 = task_a(input).result()
    else:
        step1 = task_b(input).result()
    feedback = interrupt({"draft": step1})
    return feedback
```

---

## Idempotency

Tasks should be idempotent when possible: running them multiple times with the same input should produce the same result. This is important because tasks may be re-executed during error recovery.

```python
# Idempotent: safe to retry
@task
def idempotent_save(user_id: str, data: dict) -> str:
    db.upsert(user_id, data)  # Upsert is idempotent
    return "saved"

# NOT idempotent: retry causes duplicate
@task
def not_idempotent_save(data: dict) -> str:
    db.insert(data)  # Insert creates a new row each time
    return "saved"
```

## Common Pitfalls

### Side Effects Outside Tasks

Side effects (API calls, database writes) should happen inside tasks, not in the entrypoint body. The entrypoint body re-executes on resume, but task results are cached.

```python
# BAD: Side effect in entrypoint body (runs again on resume)
@entrypoint(checkpointer=MemorySaver())
def bad_workflow(input: str) -> str:
    send_notification("Starting...")  # Sent AGAIN on resume!
    result = my_task(input).result()
    feedback = interrupt(result)
    return feedback

# GOOD: Side effect inside a task (cached, not re-run)
@task
def notify_and_process(input: str) -> str:
    send_notification("Starting...")
    return process(input)

@entrypoint(checkpointer=MemorySaver())
def good_workflow(input: str) -> str:
    result = notify_and_process(input).result()  # Cached on resume
    feedback = interrupt(result)
    return feedback
```

### Non-Deterministic Control Flow

Avoid control flow that depends on external state (time, random numbers, API results) in the entrypoint body. Use tasks for anything non-deterministic.

```python
# BAD: Time-dependent branching in entrypoint
@entrypoint(checkpointer=MemorySaver())
def bad_workflow(input: str) -> str:
    from datetime import datetime
    if datetime.now().hour < 12:  # Different on replay!
        result = morning_task(input).result()
    else:
        result = afternoon_task(input).result()
    return result

# GOOD: Time check inside a task
@task
def time_aware_process(input: str) -> str:
    from datetime import datetime
    if datetime.now().hour < 12:
        return morning_logic(input)
    return afternoon_logic(input)

@entrypoint(checkpointer=MemorySaver())
def good_workflow(input: str) -> str:
    result = time_aware_process(input).result()
    return result
```
