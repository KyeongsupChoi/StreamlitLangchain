# Memory Concepts

Memory enables agents to retain and recall information across interactions. LangGraph provides two fundamental types of memory, distinguished by scope and lifetime.

---

## Short-Term Memory

Short-term memory is **thread-scoped**: it lives within a single conversation and is discarded when the thread ends. It is the conversation history (the list of messages exchanged between the user and the agent).

LangGraph implements short-term memory through **checkpointers**. A checkpointer persists every state snapshot to a backend (in-memory, SQLite, Postgres), allowing the agent to resume mid-conversation and maintain context.

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# Each thread_id tracks a separate conversation
config = {"configurable": {"thread_id": "thread-abc"}}
result = graph.invoke(
    {"messages": [{"role": "user", "content": "My name is Alice"}]},
    config=config
)

# The agent remembers within the same thread
result = graph.invoke(
    {"messages": [{"role": "user", "content": "What is my name?"}]},
    config=config
)
# -> "Your name is Alice"
```

Short-term memory is bounded by the context window. For long conversations, apply trimming or summarization strategies to keep the message list manageable.

```python
from langchain_core.messages import trim_messages

trimmed = trim_messages(
    messages,
    max_tokens=1000,
    strategy="last",
    token_counter=model,
    allow_partial=True
)
```

## Long-Term Memory

Long-term memory persists **across sessions and threads**. It allows the agent to recall facts about a user, learn from past interactions, and adapt its behavior over time. LangGraph implements long-term memory through **stores** with a namespace/key pattern.

### Memory Types

| Type         | What It Stores              | Example                                       | Approach         |
|--------------|-----------------------------|-----------------------------------------------|------------------|
| **Semantic** | Facts and knowledge         | "User prefers dark mode"                      | Profile or Collection |
| **Episodic** | Past experiences / few-shot | "Last time user asked about billing, I used tool X and it worked" | Collection       |
| **Procedural** | Instructions and rules    | "Always greet the user by name"               | Profile          |

### Semantic Memory: Profile vs Collection

**Profile** stores a single, continuously updated document per namespace. Good for user preferences and accumulated facts.

```python
# Profile approach: one document, updated over time
store.put(("users", user_id), "profile", {
    "name": "Alice",
    "preferences": {"theme": "dark", "language": "en"},
    "tier": "premium"
})

# Retrieve the profile
item = store.get(("users", user_id), "profile")
print(item.value)
# {"name": "Alice", "preferences": {"theme": "dark", "language": "en"}, "tier": "premium"}
```

**Collection** stores many individual memory items. Good for accumulating facts, experiences, or examples that can be searched later.

```python
# Collection approach: many discrete memories
store.put(("memories", user_id), "mem-1", {"fact": "User prefers short responses"})
store.put(("memories", user_id), "mem-2", {"fact": "User is a Python developer"})
store.put(("memories", user_id), "mem-3", {"fact": "User works at Acme Corp"})

# Search for relevant memories
items = store.search(("memories", user_id), query="programming language preferences")
```

## Writing Memories

Memories can be written via two paths:

### Hot Path (Inside the Agent)

The agent writes memories as part of its normal execution flow. This is immediate but adds latency to the response.

```python
from langchain.tools import tool, ToolRuntime

@tool
def remember_preference(preference: str, runtime: ToolRuntime) -> str:
    """Remember a user preference for future conversations."""
    store = runtime.store
    user_id = runtime.context.user_id
    import uuid
    memory_id = str(uuid.uuid4())
    store.put(("preferences", user_id), memory_id, {"preference": preference})
    return f"I'll remember that: {preference}"
```

### Background (Post-Processing)

Memories are extracted and stored after the conversation completes. This avoids adding latency to the user-facing response.

```python
def post_process_memories(state, config, store):
    """Extract and store memories after the conversation ends."""
    messages = state["messages"]
    # Use an LLM to extract facts from the conversation
    facts = extract_facts(messages)
    user_id = config["configurable"]["user_id"]
    for fact in facts:
        import uuid
        store.put(("memories", user_id), str(uuid.uuid4()), {"fact": fact})
```

## Memory Storage

### InMemoryStore

`InMemoryStore` provides an in-memory implementation of the store interface. It supports optional semantic search when configured with an embedding function.

```python
from langgraph.store.memory import InMemoryStore

# Basic store without semantic search
store = InMemoryStore()

# Store with semantic search capabilities
store = InMemoryStore(
    index={
        "embed": embedding_function,  # e.g., OpenAIEmbeddings()
        "dims": 1536                  # Dimensionality of embeddings
    }
)
```

### Namespace and Key Pattern

Every memory is addressed by a **namespace** (a tuple of strings) and a **key** (a string). The namespace creates a hierarchy, and the key identifies a specific item within that namespace.

```python
from langgraph.store.memory import InMemoryStore

store = InMemoryStore(index={"embed": embed, "dims": 2})

user_id = "user-123"
application_context = "chatbot"
namespace = (user_id, application_context)

# Put a memory
store.put(namespace, "a-memory", {"rules": ["User likes short language"]})

# Get a specific memory by key
item = store.get(namespace, "a-memory")
print(item.value)
# {"rules": ["User likes short language"]}

# Search memories with filters and semantic query
items = store.search(
    namespace,
    filter={"my-key": "my-value"},
    query="language preferences"
)
```

### Persistent Backends

For production use, swap `InMemoryStore` for a persistent backend:

```python
# PostgreSQL-backed store
from langgraph.store.postgres import PostgresStore

store = PostgresStore(connection_string="postgresql://user:pass@localhost/db")

# Or use async version
from langgraph.store.postgres import AsyncPostgresStore
```

## Procedural Memory Example

Procedural memory stores instructions that govern agent behavior. A common pattern uses two nodes: one that reads the current instructions and calls the model, and another that updates the instructions based on feedback.

```python
from langgraph.graph import StateGraph, MessagesState, START, END

def call_model(state, config, store):
    """Read procedural instructions from the store and use them in the prompt."""
    user_id = config["configurable"]["user_id"]

    # Retrieve current instructions from the store
    items = store.search(("instructions", user_id), query="system prompt")
    if items:
        instructions = items[0].value["text"]
    else:
        instructions = "You are a helpful assistant."

    messages = [{"role": "system", "content": instructions}] + state["messages"]
    response = model.invoke(messages)
    return {"messages": [response]}


def update_instructions(state, config, store):
    """Update procedural memory based on conversation feedback."""
    user_id = config["configurable"]["user_id"]
    messages = state["messages"]

    # Use an LLM to generate updated instructions
    new_instructions = generate_updated_instructions(messages)
    store.put(
        ("instructions", user_id),
        "system-prompt",
        {"text": new_instructions}
    )
    return state


builder = StateGraph(MessagesState)
builder.add_node("call_model", call_model)
builder.add_node("update_instructions", update_instructions)
builder.add_edge(START, "call_model")
builder.add_edge("call_model", "update_instructions")
builder.add_edge("update_instructions", END)

graph = builder.compile(store=store)
```

This pattern allows the agent to evolve its behavior over time. As users provide feedback or corrections, the procedural memory is updated and future interactions reflect those changes.
